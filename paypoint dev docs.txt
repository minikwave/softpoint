📘 PayPoint (saum) Developer Documentation v1
1. Overview
1.1 정의

PayPoint (saum)는 Pay Worlds Payment Layer에 속하는
Application Credit Abstraction Layer이다.

PayPoint는:

자산이 아니다.

증권이 아니다.

담보가 아니다.

스테이블코인이 아니다.

PayPoint는:

사용자 소비 경험을 위한 크레딧 단위이며,
정산 자산과 분리된 Application-level credit이다.

1.2 역할

PayPoint는 다음을 수행한다:

선불 크레딧 관리

적립·사용 인터페이스

바우처·상품권 발행

가맹점 리워드 시스템

정책 기반 소비 제한

Settlement Layer와 혼합되지 않는다.

2. Architectural Position
User
  ↓
Pay Wallet
  ↓
PayPoint (saum)
  ↓
Payment Layer
  ↓

PayPoint는:

Settlement를 직접 수행하지 않는다.

유동성 풀을 관리하지 않는다.

리저브 정책을 실행하지 않는다.

14.2 Spend Algorithm (원자적 처리)
async function spend(userId: string, amount: bigint) {
  return db.transaction(async (trx) => {
    const account = await trx.selectForUpdate("paypoint_accounts", userId);

    if (account.balance < amount) {
      throw new Error("INSUFFICIENT_BALANCE");
    }

    account.balance -= amount;

    await trx.update("paypoint_accounts", account);

    await trx.insert("paypoint_transactions", {
      type: "SPEND",
      amount,
      user_id: userId
    });
  });
}

핵심: SELECT FOR UPDATE + transaction block 필수

CREATE TABLE paypoint_accounts (

  status TEXT NOT NULL DEFAULT 'ACTIVE',

15.2 paypoint_transactions
CREATE TABLE paypoint_transactions (
  tx_id UUID PRIMARY KEY,
  account_id UUID REFERENCES paypoint_accounts(account_id),

  order_id TEXT,
  receipt_id TEXT,
  metadata JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

0.2 절대 원칙

PayPoint는 자산(Asset) 이 아니라 크레딧(Credit) 이다.

PayPoint Engine은 정산/유동성/DeFi를 직접 운용하지 않는다.

Stablecoin 전환은 “사용자 UI 기능”이 아니라 Back-end Routing/Execution 기능이다.

UI(Web/Mobile/Console)는 엔진을 대체하지 않으며, 엔진의 상태를 렌더링한다.

1. 시스템 분리 아키텍처
1.1 3-컴포넌트 분리 (필수)

      ↑     ↑
      |     |
[saum WebApp/Mobile]    [Operator Console]
(UI/UX)                  (Ops/Policy/Reports)

엔진은 “단일 진실(SSOT)”이다.
웹앱/모바일/콘솔은 엔진 API만 호출한다. DB 직접 접근 금지.

6. PayPoint ↔ Stablecoin 전환 구조 (핵심)

요청하신 “전환 구조”는 세 가지로 나눠야 안전합니다.

6.1 전환은 3종류가 있다

User Cash-out Conversion

사용자가 PayPoint를 stable로 “전환/출금”하고 싶어하는 요구

규제/정책상 가장 민감

Merchant Settlement Conversion

가맹점 정산을 stable로 받고 싶다는 요구

Pay Worlds 철학과 가장 정합

Internal Treasury Conversion

운영자가 리저브/정산을 위해 내부적으로 stable을 확보/교환

Liquidity & Assets 레이어에 가까움

PayPoint 문서에는 (1)은 제한적/정책 기반, (2)(3)은 구조적으로 강하게 지원하는 톤이 맞습니다.

6.2 권장: “전환”을 UI에서 자산교환처럼 보이게 하지 말 것

PayPoint는 자산 앱이 아니므로,

스왑 화면

차트

호가

교환 비율 노출

은 최소화해야 합니다.

대신:

“정산 옵션(Stable)” 또는 “출금 옵션(허용된 경우)”
같은 표현으로 정책 기반 흐름으로 처리합니다.

6.3 Conversion Architecture (권장 설계)

PayPoint Engine 자체가 스왑을 하지 않습니다.
Conversion Router를 별도 컴포넌트로 둡니다.

PayPoint Engine
  - create ConversionRequest
  - lock credits (reserved_balance)
        ↓
Conversion Router (Execution Service)
  - price quote
  - routing (CEX/DEX/OTC)
  - tx execution
        ↓
Settlement Receipt update
  - COMMITTED → SETTLED
  - record txHash / settlementRef
구성 요소

ConversionRequest (엔진에 기록)

Quote Provider (가격/수수료)

Route Executor (DEX/CEX/OTC)

Settlement Adapter (PayChain/IEUM 연동)

6.4 ConversionRequest 상태 머신
REQUESTED
  ↓ (policy pass)
AUTHORIZED (lock credits)
  ↓ (execution started)
EXECUTING
  ↓
SETTLED (txHash recorded)
  ↘
FAILED (unlock credits)
설계 포인트

REQUESTED는 UI 요청

AUTHORIZED에서 credit lock

FAILED 시 credit unlock

SETTLED 시 credit burn/consume 확정

6.5 정책 기반 전환 제한 (필수)

Conversion은 규제 민감도가 높으므로, PayPoint 엔진의 Policy Engine에서 반드시 통제합니다.

18. Conversion Router 개요
18.1 정의

Conversion Router는 다음을 수행한다:

PayPoint 크레딧을 Stablecoin으로 전환하거나
Stablecoin 기반 정산을 실행하기 위한 실행 계층

Conversion Router는:

PayPoint Engine과 분리된다.

Settlement Layer와 직접 결합되지 않는다.

Liquidity & Assets Layer와 연결된다.

18.2 구조
PayPoint Engine
   ↓ (ConversionRequest)
Conversion Router
   ↓ (Route Execution)
External Liquidity (DEX / CEX / OTC)
   ↓
Settlement Confirmation
   ↓
PayPoint Engine update (SETTLED / FAILED)
