## 3. “포인트–은행 / 포인트–온체인 / PCI 잔고” 모니터링·비교 인프라

### 3.1 결론

**본 레포에는 해당 인프라가 없다.** 다음이 확인되었다.

- **은행(법정화폐) 보관 잔고**·**신탁/분리보관 계정** 스냅샷을 저장하거나, 엔진 내 **총 사용자 부채(`sum(balance)`)** 와 비교하는 **테이블·잡·API·대시보드가 없음**.
- **온체인** 실제 토큰/스테이블 잔고를 조회해 **오프체인 기록(PayPoint 합계·전환 예약 등)** 과 맞추는 **리컨실리에이션(reconciliation) 워커·알림이 없음**. 전환은 `tx_hash`·`settlement_ref` 필드만 있고, 체인 RPC 연동 코드 없음.
- **PCI(카드 결제)** 로 들어온 자금·정산 잔고와 **발행된 PayPoint(충전분)** 를 맞추는 **PG 웹훅 대사·미수·충당 모델** 없음. `PG-3`는 로드맵(§10.5)만 존재.
- **`apps/paypoint-worker`** 디렉터리 **미존재**. 배치·리포트·주기적 대사는 구조상 권장만 되어 있음.
- **Prometheus**: `ENABLE_PROMETHEUS_METRICS=true` 일 때만 `GET /metrics` 노출(의도적으로 기본 비활성). **Grafana 대시보드·알림·은행·체인·PG 대사용 메트릭** 은 여전히 없음.
- **OpenTelemetry** 등 분산 트레이싱 **없음**. `GET /health` 는 정적 OK 응답 수준.

### 3.2 엔진 DB가 갖는 것(참고)

- **사용자 계정별** `balance`·`reserved_balance` 및 `paypoint_transactions` — **사용자에게 보이는 크레딧의 SSOT** 로만 역할이 정의되어 있음.
- **은행 원장·PCI 입금·온체인 커스터디** 를 표현하는 별도 엔티티는 스키마에 없음.

### 3.3 나중에 “완비”하려면 보통 필요한 층(제품·컴플라이언스 별도 설계)

1. **외부 진실 소스** 스냅샷: 은행 API/수동 업로드, 체인 RPC/인덱서, PG 정산 파일·웹훅.
2. **내부 집계**: 시점별 `SUM(paypoint_accounts.balance)` (및 reserved, conversion 상태별).
3. **대사 규칙·허용 오차·드리프트 알림**(슬랙/페이저).
4. **감사 추적**: 스냅샷 ID, 대사 실행 시각, 불일치 티켓(예외 큐와 연동 가능).
5. **PCI 범위**: 카드 데이터는 엔진 밖; 충전 서비스가 `issue` 만 호출하는 패턴과 **재무 대사**는 별도 시스템이 흔함.

이 레포는 **애플리케이션 크레딧 엔진 MVP** 에 집중되어 있으며, **트레저리·대사·은행/체인/PG 모니터링** 은 **추가 프로젝트**로 두는 것이 현재 상태와 일치한다.
